# RFC-0003 — `tide repl` and reflection API

| Field | Value |
|---|---|
| Number | 0003 |
| Status | draft |
| Created | 2026-05-27 |
| Supersedes | — |
| Target | new `cmd/tide` subcommand `repl`; entry-point table in `README.md`; **language-surface**: a `reflect` module exposed through the binding surface, with paired edits in `lang-spec/builtins.md` and `lang-spec/type-system.md` (introduction of `any`) |

## Summary

Two paired user-visible additions:

1. **`tide repl`** — an interactive prompt. The user types Tide
   source one input at a time and sees the result. The grammar
   accepted at the prompt is the same `.td` grammar, with one
   relaxation: top-level declarations, `let`/`var` bindings,
   assignments, and bare expressions are all valid inputs (a
   synthetic `main` is implicit).
2. **`reflect` module** — a small runtime-reflection API.
   Without it the REPL is unable to answer the basic questions
   that motivate having a REPL in the first place
   ("what type is this?", "what fields does this object have?"),
   and Tide instances pretty-print as opaque Go-shaped values.
   The API is callable from any Tide program; the REPL just
   happens to be its first heavy consumer.

This RFC describes the **user-visible contract** for both.
Internal implementation choices (interpreter strategy, runtime
type-metadata layout, session-source representation) belong in
`docs/architecture.md` once they exist and are not normative
parts of this RFC.

## Motivation

Two problems we currently cannot solve:

1. **No way to "just try" a piece of Tide.** Tide has no
   interactive entry. Every experimentation requires editing a
   `.td` file. With PR-F4 we now have classes — exactly the
   construct people want to try interactively — and no way to
   try them. The TypeScript developers the language targets
   reach for `node` / `deno repl` reflexively; we should give
   them the same reach.

2. **No way to interrogate a value at runtime.** Even outside
   the REPL, the user has no way to ask "what is the type of
   this value?" or "iterate over the fields of this record".
   Today, printing a class instance produces something like
   `&{0}` — a Go-shaped dump that loses every Tide-side name.
   Once sema is in place, static introspection (`typeof`,
   exhaustive `match`) will cover some of this. Runtime
   reflection covers the rest: heterogeneous containers,
   tooling, serialisers, pretty-printers.

These two are coupled because the REPL is the API's first
non-trivial consumer. A meta-command like `:inspect c` and a
value-friendly auto-printer both go through `reflect`, and they
are the things a user expects to find when they sit down at a
REPL.

## Design — `tide repl`

### Invocation

```
tide repl
```

No file arguments. No flags in v1. Reads stdin, writes
stdout / stderr. Exits 0 on clean EOF (`Ctrl-D` or `:quit`),
non-zero only on internal failure — runtime errors *in user
inputs* never affect the REPL's own exit status.

### Prompt

| State | Prompt |
|---|---|
| Ready for a fresh input | `tide> ` |
| Continuing a multi-line input | `... > ` |

### What the REPL accepts

The REPL parses each input with the same parser as `tide run`.
The grammar is unchanged. The one relaxation: at the outermost
level of an input the user can write things that, in a `.td`
file, would have to live inside `func main()`.

An input is one of:

| Shape | Effect |
|---|---|
| `import <path>` | Make the import available to all later inputs. |
| `func`, `type`, `class` declaration | Declare it; later inputs see it. Re-declaring a name shadows the prior definition. |
| `let <pat> = <expr>` / `var <name> = <expr>` | Bind a name; later inputs read it (and assign to a `var`). Re-binding shadows. |
| `<lvalue> = <expr>` | Mutate a prior `var`. |
| Bare `<expr>` | Evaluate and pretty-print the result (see "Auto-printing"). The session is unchanged. |

Rejected at the outermost level of an input in v1:

- `if`, `for`, `while`, `match`. They remain valid inside a
  function, method, or block the user defines at the prompt.
  Error: `repl: top-level control-flow not supported in v1 — wrap it in a func`.
- User-typed `func main()`. The REPL owns `main`. Error:
  `repl: main is owned by the REPL`.
- `return`, `break`, `continue` at the outermost level —
  meaningless without an enclosing function or loop.

The principle: the input grammar is the same `.td` grammar.
The only relaxation is "the synthetic `main` is implicit".

### Auto-printing

A bare expression input prints its value to stdout. The printer
uses the `reflect` API (see below) to render values
type-aware:

- Primitives — Go-style (`42`, `"hello"`, `true`).
- Sum variants — `ColorRed` prints as `Red` (the Tide-side
  name).
- Classes — `Counter{n: 3}`, named fields in declaration order.
- Slices — `[1, 2, 3]` with Tide-side elements.

A `let`/`var` input does **not** print (binding is silent). A
declaration input does not print. The result of an expression
evaluating to `unit` prints an empty line (known sharp edge,
acceptable for v1).

### Multi-line input

An input is submitted when it is balanced: all `{` `(` `[` are
matched, no string literal is open mid-line, and the line does
not end with a continuation-implying token (trailing comma,
binary operator, open paren). Until then, the continuation
prompt reads more lines. `Ctrl-C` on an empty continuation
line abandons the buffer and returns to the fresh prompt.

### Errors

| Class | What the user sees | Session effect |
|---|---|---|
| Parse error | `repl:line:col: error[E0NNN]: <message>`. | None — input discarded. |
| Compile error (codegen / sema once it lands) | Same format. | None — input discarded. |
| Runtime error in the user's program | The program's stderr is shown verbatim. | A prior successful `let`/`var`/declaration *in the same input* is kept (the binding happened); the failing expression bound no name. |

### Meta-commands

Lines beginning with `:` are interpreted by the REPL.

| Command | Effect |
|---|---|
| `:help` | Print the meta-command list. |
| `:quit` / `:q` | Exit. |
| `:reset` | Drop all session state. |
| `:imports` | Print the current import list. |
| `:show` | Print the current session source (everything the user has typed that the REPL kept). Diagnostic aid. |
| `:type <expr>` | Print the static type of `<expr>` via `reflect.typeName`. Requires `<expr>` to be evaluable in the current session. |
| `:inspect <expr>` | Pretty-print `<expr>` with all fields enumerated via `reflect.fields`. For a class / record, recursive one level by default. |
| `:load <file.td>` | Read `<file.td>` and process its top-level forms as if typed at the prompt. The file must not contain `func main()`. |

Unrecognised `:`-prefixed line → `repl: unknown meta-command :foo`.
No Tide expression in v1 begins with `:`, so the namespace is
unambiguous.

### Out of scope for v1

Each item below is its own follow-up RFC if it becomes
important enough.

- Arrow-key history, in-line editing, history-file persistence.
- Tab completion.
- Async / spawn orchestration across inputs.
- Notebook-style UI.
- Colour output.

## Design — `reflect` module

The `reflect` module is a normal Tide module — users access it
with `import reflect`. Its v1 surface is small and chosen by
what the REPL's auto-printer and meta-commands need.

### Types

```
// A first-class runtime type descriptor. Two Type values are
// equal iff they describe the same Tide type.
type Type = ...   // opaque

// Categorisation of what kind of type a Type describes.
type Kind =
  | Primitive
  | Class
  | Sum
  | Slice
  | Function
  | Unit

// A single field of a class or record.
type FieldInfo = {
  name: string,
  type: Type,
}

// A method of a class.
type MethodInfo = {
  name: string,
  is_static: bool,
}

// A variant of a sum type.
type VariantInfo = {
  name: string,
  tag: int,             // declaration-order tag (per lowering-go.md)
}
```

### `any`

To pass values of unknown type to `reflect`, this RFC introduces
the `any` type. `any` is the top type — every value
assignable to `any`. The compiler lowers `any` to Go's
`interface{}`. `any` cannot be the static type of a function
parameter outside `reflect`-flavoured contexts in v1 — type
rules in `lang-spec/type-system.md` constrain it to (a)
parameter positions of functions in the `reflect` module and
(b) any field annotated `any` explicitly. This restriction
exists to prevent `any` from becoming a `TypeScript`-style
gradual-typing escape hatch by accident.

(`any` as a name is established practice; the alternative
`Any` is also acceptable. Decision: lowercase `any`, matching
the closed `PrimitiveName` shape — but `any` is technically
non-primitive. Flagged in Open Questions.)

### Functions

```
func typeOf(v: any): Type
func typeName(t: Type): string         // e.g., "Counter", "int", "Option<int>"
func kind(t: Type): Kind

// Class and record introspection
func fields(t: Type): []FieldInfo      // empty for non-class/record types
func methods(t: Type): []MethodInfo    // empty for non-class types
func fieldValue(v: any, name: string): any   // panics if no such field

// Sum-type introspection
func variants(t: Type): []VariantInfo  // empty for non-sum types
func variantOf(v: any): VariantInfo    // panics if v is not a sum value
```

The API is deliberately minimal: enough to write a generic
pretty-printer, an interactive inspector, and the REPL's
`:type` / `:inspect`. Mutation through reflection
(`setField`, `callMethod`) is **not** in v1; it adds a much
larger semantics problem and unblocks no immediate user need.

### Use site sketch (informative)

```tide
import reflect

func dump(v: any): string {
  let t = reflect.typeOf(v)
  match reflect.kind(t) {
    Primitive => fmt.sprintf("%v", v),
    Class => {
      var out = reflect.typeName(t) + "{"
      for f in reflect.fields(t) {
        out = out + f.name + ": " + dump(reflect.fieldValue(v, f.name)) + ", "
      }
      return out + "}"
    },
    // ...
  }
}
```

(The above is illustrative — not a normative example. `match`
on `any`-typed values requires the type rules in
`lang-spec/type-system.md` to admit it; the paired edit covers
this.)

### Performance and identity

This RFC makes no promises about `Type` value identity outside
equality. Whether `typeOf(a) == typeOf(b)` is `O(1)` is an
implementation concern. Two `Type` values *are* equal iff they
describe the same Tide type — this is the only externally
observable invariant.

## Alternatives considered

1. **Compile-time `typeof` operator instead of a runtime
   `reflect` module.** `typeof x` would yield a static type
   description, like TypeScript's `typeof`. *Rejected for v1.*
   Static `typeof` requires sema; sema is Phase 2. Static
   `typeof` also doesn't help the REPL pretty-printer, which
   has *values* in hand and wants to ask "what type is this *at
   runtime*?" — a different question.
2. **Reflection as a compiler magic-builtin set, not a
   user-importable module.** `__typename__(v)` style. *Rejected.*
   Tide is committed to the principle that things accessible to
   the language live in normal modules (D6). The binding
   surface already accommodates user-facing modules; `reflect`
   slots in naturally.
3. **Notebook-style cells instead of a line-based REPL.**
   *Rejected for v1.* Higher UI effort; does not change what
   the user can *do*. Can layer on top of the line-based REPL
   later.
4. **Auto-print without reflection — call `fmt.println(v)` for
   the value and accept Go-shaped output.** *Considered.* This
   is what a "REPL without reflection" looks like — and it's
   exactly the unsatisfying experience this RFC is meant to
   avoid. The cost of *not* having reflection is permanently
   visible in every interactive session.

## Paired edits

- `lang-spec/type-system.md` — introduce `any` with restricted
  use (see "any" above). New typing rules T-Any-Conv,
  T-Any-Field.
- `lang-spec/builtins.md` — describe the `reflect` module's
  surface: `Type`, `Kind`, `FieldInfo`, `MethodInfo`,
  `VariantInfo`, and the listed functions. Tag `reflect` as a
  runtime-supplied module (not a Go-stdlib binding).
- `lang-spec/diagnostics.md` — new diagnostic codes for
  `reflect.fieldValue` / `reflect.variantOf` panics on
  mismatched shape; for REPL-specific input rejections
  (top-level control-flow, user-typed `main`).
- `README.md` — `tide repl` row in the entry-point table.

No `lang-spec/lowering-go.md` change is *normatively* required
by this RFC; how `Type` descriptors are produced and stored at
runtime is the implementation's call. (For implementers'
context: a small per-class / per-sum metadata table at codegen
time, attached to a runtime registry, is the obvious shape.
This belongs in `docs/architecture.md` when the implementing
PR lands.)

## Transition / compatibility

Not applicable. `tide repl` is a new subcommand. `reflect` is a
new module. `any` is a new type that nothing currently
mentions. All purely additive.

## Open questions

1. **`any` casing.** Lowercase `any` mirrors `PrimitiveName`
   shape and TypeScript's `any` keyword (the muscle memory the
   language targets). Capital `Any` mirrors Tide's PascalCase
   user-type convention. Bias toward lowercase.
2. **Scope of `any`.** This RFC restricts `any` to (a) `reflect`
   parameter positions and (b) explicitly annotated fields.
   Should we also admit `any` in collections (`[]any`,
   `Map<string, any>`)? The user-need is real (heterogeneous
   container) but slippery (gradual-typing risk). Deferred.
3. **`:type <expr>` on a not-yet-bound name.** If the user
   types `:type x` and `x` is unknown, do we just say "no such
   binding" or try to parse the input as a fragment and report
   what *would* be its type? Bias: simple "no such binding".
4. **Reflection on closures / function values.** v1 says
   `kind(t) == Function` is possible but offers no further
   introspection (no `params`, no `return_type`). Closures
   make this messier still. Deferred until a user need lands.
5. **REPL behaviour for prior let-initialiser side effects.**
   If `let now = time.Now()` is typed once, subsequent inputs
   reading `now` should see the value from the original
   evaluation, not re-evaluate the initialiser. This is
   user-observable; the implementing PR must preserve this
   invariant regardless of internal strategy.
