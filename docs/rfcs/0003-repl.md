# RFC-0003 — `tide repl` (accumulating-source mode)

| Field | Value |
|---|---|
| Number | 0003 |
| Status | draft |
| Created | 2026-05-27 |
| Supersedes | — |
| Target | new `cmd/tide` subcommand (`repl`), new `internal/repl` package; **no** `lang-spec/` changes (REPL is a tooling concern, not a language-surface change) |

## Summary

Add an interactive subcommand `tide repl` that lets the user type
Tide source one line (or one paste-block) at a time, evaluate it
against an accumulating session, and see the result printed. The
v1 implementation has **no separate interpreter**: every input
re-compiles a synthetic `.td` "session source" through the
existing `lexer → parser → codegen → go run` pipeline. State —
imports, type/class declarations, top-level let/var bindings —
persists across inputs because each input prepends the prior
session source. This is by design the slowest reasonable
implementation (~50–200 ms per input on the Go toolchain
cold-path), traded for *zero* divergence between `tide run` and
`tide repl` semantics.

## Motivation

Tide today has two entry points: `tide build` and `tide run`,
both file-driven. There is no way to ask "what does this
expression evaluate to?" or "is this snippet syntactically
well-formed?" without writing a `.td` file and invoking the
compiler.

This matters now, not later, because:

1. **Examples in `examples/` are write-only.** We have 50+ Tide
   programs in the corpus; touching any of them to "just try a
   thing" means editing the file and re-running. A REPL is the
   shortest path to experimentation, especially for the
   TypeScript-developer audience the language targets — `node`
   and `deno repl` are part of their muscle memory.
2. **PR-F4 ships classes.** A user kicking the tires on classes
   wants to type `let c = Counter.make(0); c.bump(); c.get()`
   and see `1`. Compelling a write-a-file ritual for that is
   friction we can erase in one PR.
3. **Acceptance-suite triage.** When an example breaks, the
   first move is "reduce it to a minimal repro". A REPL lets us
   bisect interactively instead of editing scratch files.

## Design

### Subcommand shape

```
tide repl [--no-color] [--history <path>]
```

No file arguments. Reads stdin line-by-line (or block-by-block
on paste), writes a prompt and the input's stdout/stderr to
the terminal. Exits on EOF (`Ctrl-D`) or `:quit`.

The prompt is `tide> ` for a fresh input and `... > ` (with a
trailing space) for continuation lines (see "Multi-line input"
below).

### Session model — accumulating source

A REPL session maintains a single in-memory **session source**:
a Tide source string that grows as the user submits inputs.

- **Imports** and **top-level declarations** (`func`, `type`,
  `class`, top-level `let`/`var`) are appended to the session
  source in declaration order.
- **Statements at "value position"** (an expression alone, or a
  `let`/`var` the user typed at the prompt) are wrapped in a
  synthetic `func main()` regenerated fresh on each input. Prior
  top-level state is reused; prior `main()` is discarded.
- The synthetic `main` ends with a `fmt.println` of the input's
  trailing expression when one exists (so `1 + 1` prints `2`,
  `let x = 1 + 1` prints nothing).

After classifying the input, the runtime constructs:

```
<session imports>

<session top-level declarations>

func main() {
  <prior session lets and vars, in order>
  <this input's body>
  // synthesized only if the input has a trailing expression:
  fmt.println(<that expression>)
}
```

This source is written to a temp file and run via the existing
`cmd/tide run` machinery (`go run` under the hood, **not** a
shelled-out `tide run` — we call `codegen.Emit` + `goRun`
directly to avoid a process round-trip). The REPL prints
stdout / stderr verbatim.

### Classification per input

The REPL parses each input with the **same** `internal/parser`
used by the compiler and routes by AST shape:

| Input AST root | Effect on session |
|---|---|
| `Import` | Append to session imports. No execution. |
| `FuncDecl`, `TypeDecl`, `ClassDecl` | Append to session top-level decls. No execution (unless the input also contains a stmt; rejected for v1). |
| `LetStmt` (top-level position) | Append the **declaration** to session top-level lets; run `main()` if the let's value has side effects (print, etc.) — for v1 we always re-run `main()`. |
| `VarStmt` (top-level position) | Same as `LetStmt`. |
| `AssignStmt` | Treated as session-let-mutation: append to session "post-let stmts" list and re-run. Without sema this means the parser accepts assignments to prior `var x`. |
| `ExprStmt` (a plain expression) | Wrap in `func main()` with a trailing `fmt.println(<that expr>)` and run. Do **not** modify session source. |
| Anything else (e.g., `if`, `for` at top level) | For v1 reject with `repl: only declarations, let/var, assignment, and expressions are accepted at the prompt`. Same wrapping pattern can be extended later. |

This routing keeps the REPL's grammar identical to the
language's. The user does not get a "REPL dialect" that allows
floating statements anywhere — they get the existing grammar
with a synthetic `main` wrapper for the value-position cases.

### Multi-line input

The REPL reads until the input is **balanced** — equal counts of
opening and closing `{` `(` `[` brackets, no open `"` string,
no half-finished `\` escape, no trailing `,` at line break. The
heuristic lives in `internal/repl/balance.go` and uses the
existing `internal/lexer` (lex the input prefix; if the lexer
reports unterminated structure, prompt for more).

On a balanced submission the REPL classifies and runs.

### Meta-commands

A short set of colon-prefixed commands, none of which touch the
language grammar:

| Command | Effect |
|---|---|
| `:help` | Print the meta-command list. |
| `:quit` / `:q` | Exit. Same as EOF. |
| `:reset` | Drop the session source. Imports and declarations gone. |
| `:imports` | Print the current session import list. |
| `:show` | Print the current session source (what we'd send to `codegen.Emit`). For debugging. |
| `:load <file.td>` | Read a `.td` file, classify its top-level forms as if typed at the prompt. Useful for picking up a prelude. |

If a line begins with `:` and matches none of the above, it's
an error (`repl: unknown meta-command :foo`). If a line begins
with `:` but the user wanted a Tide expression starting with
`:` — there is no such expression in v1, so the namespace is
unambiguous.

### Error handling

Three classes of error, all surface at the prompt without
killing the session:

1. **Parse error** — `internal/parser` returns a `*Diag`. Print
   the diagnostic with `.td` coordinates (`repl input:line:col:
   error[E0NNN]: <message>`). The session source is **not**
   mutated.
2. **Codegen error** — `internal/codegen.Emit` returns a
   non-nil error. Same treatment as parse errors. Session source
   **not** mutated (so the bad input doesn't poison the next
   one).
3. **Runtime error from the user program** — `go run` exits
   non-zero. The REPL prints whatever the program emitted on
   stderr, then returns to the prompt. The session source **is**
   committed in this case (because the program ran — the user
   asked for execution, the execution happened, it just
   produced an error). If the user wants to undo, `:reset`.

Two-phase commit on the session source: classify+parse first,
mutate the session source only if the parse succeeded, **then**
run. A `go run` failure post-classification keeps the new state.

### Output capture

`fmt.println` and friends go to the REPL's stdout. There is no
prompt-injection risk because the REPL writes its own `tide> `
prompt only after the child process has terminated.

The REPL does **not** intercept stdout to format value-position
results specially — `1 + 1` prints `2\n` because the synthetic
`fmt.println(1 + 1)` prints `2\n`. This is intentionally less
fancy than Node's `> 2` styling: same code path as a `.td`
program, no formatter to drift.

### What's *not* in v1

- No history file (despite the `--history` flag in the synopsis —
  v1 ignores it; the flag is reserved). `readline`-style line
  editing is provided by `golang.org/x/term` or a tiny
  hand-rolled buffer; arrow-key history is **out of scope** for
  this RFC. Reserved for a follow-up once we know what people
  want.
- No tab completion. The session source doesn't have a "scope
  table" — only the AST as text. Completion would need sema, and
  sema is Phase 2.
- No `:type <expr>` — same blocker (sema).
- No async / spawn support beyond what compiled programs do
  today. A spawn-containing input runs in its own short-lived
  process; the REPL does not orchestrate goroutines across
  inputs.

## Alternatives considered

1. **Embed yaegi (Go interpreter) as the REPL backend.** Lower
   Tide to Go, hand the Go to yaegi, get incremental execution
   without spawning subprocesses. *Rejected for v1.* Adds a
   third-party dependency that owns Go-language semantics in
   parallel to `cmd/tide`'s `go run`. Any divergence (generics
   edge case, reflect, unsafe-shaped bindings) becomes a "works
   in `tide run` but breaks in `tide repl`" bug class. The
   accumulating-source approach is slower but uses **the same
   binary** the user already trusts.
2. **Write a native interpreter over the AST.** A
   tree-walking interpreter in `internal/repl/eval.go` that
   evaluates Tide directly, bypassing Go. *Rejected for v1.*
   Too much surface — every construct the language adds
   (PR-F2 sum types, PR-F3 slices, PR-F4 classes, Phase 2 `scope`)
   needs an evaluator case in addition to a codegen case. The
   bug surface grows linearly with the language. Worth
   revisiting if REPL latency becomes the bottleneck.
3. **Per-input subprocess invocation of `tide run`.** Each input
   shells out to `tide run` on a temp file built from session
   source. *Rejected.* This is what the chosen design does
   semantically, but going via subprocess adds a process spawn
   and re-parses session source on every input. Calling
   `codegen.Emit` + `go run` directly from the REPL keeps the
   Go-toolchain cost without the `exec` overhead.
4. **Persistent compiled session — incremental codegen.** Keep
   a compiled `.so` plugin around, re-link new inputs into it.
   *Rejected.* Go's `plugin` package is platform-restricted and
   notoriously fragile across builds. Engineering cost
   disproportionate to the value of dropping ~100 ms.

## Paired edits

None in `lang-spec/`. The REPL is a tooling concern; it adds no
keywords, no grammar productions, no type rules, no
diagnostics, no IR nodes, no lowering cases.

Updated documentation, all in `docs/`:

- `docs/architecture.md` — new "REPL" subsection under
  Tooling, describing the session-source model and pointing at
  `internal/repl`.
- `README.md` — add `tide repl` to the entry-point table once
  implemented.

## Transition / compatibility

Not applicable. `tide repl` is purely additive — a new
subcommand that did not exist. No file format changes, no
flag-name conflicts (`build`, `run`, `emit` continue unchanged).

## Open questions

1. **How does the REPL handle a `func main()` typed by the user?**
   The session already synthesises one. Two options: (a) reject
   user-typed `main` with a clear diagnostic ("`main` is owned
   by the REPL — drop the wrapper"); (b) accept it and replace
   our synthesised one with theirs, then `:reset` is the way
   out. Bias toward (a) for v1 to keep the model simple.
2. **What's the latency budget?** Accumulating source compiles
   the *whole* session on every input. A long session of 200
   declarations re-compiles 200 declarations every keystroke
   submission. Anecdotally `go build` on a one-file program
   ranges 50–200 ms; with 200 simple decls it might stretch to
   ~500 ms. If users complain, the v2 option is a "freeze the
   session into a `.go` binary, then `go run` only the delta" —
   but that's a separate RFC.
3. **Where do `import`s of stdlib packages with side effects
   sit?** None in the existing binding surface today, but if /
   when we bind `os/signal` or similar, repeatedly importing
   them across REPL inputs might produce surprising behaviour
   (signal handlers re-installed each input). Deferred — bind
   first, observe, then decide.
4. **Should `:load <file.td>` validate the file is REPL-shaped
   (no `func main()`)?** If the file contains a `main`, see Open
   Question 1. Probably the same answer.
