## Summary

Two paired user-visible additions:

1. **`tide repl`** — an interactive prompt. The user types Tide
   source one input at a time and sees the result. The grammar
   accepted at the prompt is the same `.td` grammar, with one
   relaxation: top-level declarations, `let`/`var` bindings,
   assignments, and bare expressions are all valid inputs (a
   synthetic `main` is implicit).
2. **`reflect` module** — a small runtime-reflection API.
   Without it the REPL is unable to answer the basic questions
   that motivate having a REPL in the first place
   ("what type is this?", "what fields does this object have?"),
   and Tide instances pretty-print as opaque Go-shaped values.
   The API is callable from any Tide program; the REPL just
   happens to be its first heavy consumer.

This RFC describes the **user-visible contract** for both.
Internal implementation choices (interpreter strategy, runtime
type-metadata layout, session-source representation) belong in
`docs/architecture.md` once they exist and are not normative
parts of this RFC.

## Motivation

Two problems we currently cannot solve:

1. **No way to "just try" a piece of Tide.** Tide has no
   interactive entry. Every experimentation requires editing a
   `.td` file. With PR-F4 we now have classes — exactly the
   construct people want to try interactively — and no way to
   try them. The TypeScript developers the language targets
   reach for `node` / `deno repl` reflexively; we should give
   them the same reach.

2. **No way to interrogate a value at runtime.** Even outside
   the REPL, the user has no way to ask "what is the type of
   this value?" or "iterate over the fields of this record".
   Today, printing a class instance produces something like
   `&{0}` — a Go-shaped dump that loses every Tide-side name.
   Once sema is in place, static introspection (`typeof`,
   exhaustive `match`) will cover some of this. Runtime
   reflection covers the rest: heterogeneous containers,
   tooling, serialisers, pretty-printers.

These two are coupled because the REPL is the API's first
non-trivial consumer. A meta-command like `:inspect c` and a
value-friendly auto-printer both go through `reflect`, and they
are the things a user expects to find when they sit down at a
REPL.

## Design — `tide repl`

### Invocation

```
tide repl
```

No file arguments. No flags in v1. Reads stdin, writes
stdout / stderr. Exits 0 on clean EOF (`Ctrl-D` or `:quit`),
non-zero only on internal failure — runtime errors *in user
inputs* never affect the REPL's own exit status.

### Prompt

| State | Prompt |
|---|---|
| Ready for a fresh input | `tide> ` |
| Continuing a multi-line input | `... > ` |

### What the REPL accepts

The REPL parses each input with the same parser as `tide run`.
The grammar is the same — every input must be a sequence of
constructs the existing parser accepts. The REPL relaxes one
*placement* rule: at the outermost level of an input the user
may write constructs that, in a `.td` file, would have to live
inside a function body (`let`, `var`, assignments, bare
expressions).

An input is one of:

| Shape | Effect |
|---|---|
| `import <path>` | Make the import available to all later inputs. |
| `func`, `type`, `class` declaration | Declare it; later inputs see it. Re-declaring a name shadows the prior definition. |
| `let <pat> = <expr>` / `var <name> = <expr>` | Bind a name; later inputs read it (and assign to a `var`). Re-binding shadows. |
| `<lvalue> = <expr>` | Mutate a prior `var`. |
| Bare `<expr>` | Evaluate and pretty-print the result (see "Auto-printing"). The session is unchanged. |

Rejected at the outermost level of an input in v1:

- `if`, `for`, `while`, `match`. They remain valid inside a
  function, method, or block the user defines at the prompt.
  Error: `repl: top-level control-flow not supported in v1 — wrap it in a func`.
- User-typed `func main()`. The REPL runs each input as if it
  were a program; declaring `main` at the prompt collides with
  that. Error: `repl: main is owned by the REPL`.
- `return`, `break`, `continue` at the outermost level —
  meaningless without an enclosing function or loop.

The principle: the input grammar is the `.td` grammar; the only
relaxation is *placement* — function-body-only constructs are
admitted at the outermost level of an input.

### Auto-printing

A bare expression input prints its value to stdout. The printer
uses the `reflect` API (see below) to render values
type-aware:

- Primitives — Go-style (`42`, `"hello"`, `true`).
- Sum variants — `ColorRed` prints as `Red` (the Tide-side
  name).
- Classes — `Counter{n: 3}`, named fields in declaration order.
- Slices — `[1, 2, 3]` with Tide-side elements.

A `let`/`var` input does **not** print (binding is silent). A
declaration input does not print. The result of an expression
evaluating to `unit` prints an empty line (known sharp edge,
acceptable for v1).

**Last-value bindings.** After a bare-expression input prints,
the value is also bound to a special identifier `_` for use by
the next input. If a runtime error fires, the error value is
bound to `_error`. Both are reset by `:reset`. Inspired by
Python / Deno; the muscle memory carries.

`_` already exists in Tide as a wildcard token in binding /
pattern positions; in expression position it is currently an
ordinary identifier (per `lang-spec/keywords.md`). The REPL
co-opts that expression-position identifier as the last-value
binding — this is a binding choice the REPL makes for its own
session scope and does not change the language's grammar.
Outside the REPL, `_` in expression position remains an
unbound name (`E0301 Unresolved identifier`).

### Multi-line input

An input is submitted when it is balanced: all `{` `(` `[` are
matched, no string literal is open mid-line, and the line does
not end with a continuation-implying token (trailing comma,
binary operator, open paren). Until then, the continuation
prompt reads more lines. `Ctrl-C` on an empty continuation
line abandons the buffer and returns to the fresh prompt.

### Errors

| Class | What the user sees | Session effect |
|---|---|---|
| Parse error | `repl:line:col: error[E0NNN]: <message>`. | None — input discarded. |
| Compile error (codegen / sema once it lands) | Same format. | None — input discarded. |
| Runtime error in the user's program | The program's stderr is shown verbatim. | A prior successful `let`/`var`/declaration *in the same input* is kept (the binding happened); the failing expression bound no name. |

### Meta-commands

Lines beginning with `:` are interpreted by the REPL.

| Command | Effect |
|---|---|
| `:help` | Print the meta-command list. |
| `:quit` / `:q` | Exit. |
| `:reset` | Drop all session state. |
| `:imports` | Print the current import list. |
| `:show` | Print the current session source (everything the user has typed that the REPL kept). Diagnostic aid. |
| `:write <file.td>` | Spill the current session source to `<file.td>`. Refuses to overwrite an existing file (error `repl: <file.td> already exists — use :write! to overwrite`); `:write! <file.td>` overwrites. The exit-ramp from interactive exploration to a saved program. |
| `:type <expr>` | Print the static type of `<expr>` via `reflect.typeName`. Requires `<expr>` to be evaluable in the current session. |
| `:inspect <expr>` | Pretty-print `<expr>` with all fields enumerated via `reflect.fields`. For a class / record, recursive one level by default. |
| `:load <file.td>` | Read `<file.td>` and process its top-level forms as if typed at the prompt. The file must not contain `func main()`. |

**Tide-terminology invariant.** `:type` output and every
reflection-driven string from the REPL (auto-printer field
names, `:inspect` headers, error coordinates) speak Tide types
in Tide spelling. Class fields appear with their declared
names, sum variants with their declared names, primitives with
the Tide spelling (`int`, not `int64`). Raw Go-side names
(`*main.Counter`, `interface{}`) never leak to the prompt. This
extends D10 (errors in `.td` coordinates and terminology) to
the REPL's interactive output.

Unrecognised `:`-prefixed line → `repl: unknown meta-command :foo`.
No Tide expression in v1 begins with `:`, so the namespace is
unambiguous.

### Out of scope for v1

Each item below is its own follow-up RFC if it becomes
important enough.

- Arrow-key history, in-line editing, history-file persistence.
- Tab completion.
- Async / spawn orchestration across inputs.
- Notebook-style UI.
- Colour output.

## Design — `reflect` module

The `reflect` module is a normal Tide module — users access it
with `import reflect`. Its v1 surface is small and chosen by
what the REPL's auto-printer and meta-commands need.

### Types

```
// A first-class runtime type descriptor. Two Type values are
// equal iff they describe the same Tide type.
type Type = ...   // opaque

// Categorisation of what kind of type a Type describes.
type Kind =
  | Primitive
  | Class
  | Sum
  | Slice
  | Function
  | Unit

// A single field of a class or record.
type FieldInfo = {
  name: string,
  type: Type,
}

// A method of a class.
type MethodInfo = {
  name: string,
  is_static: bool,
}

// A variant of a sum type.
type VariantInfo = {
  name: string,
  tag: int,             // declaration-order tag (per lowering-go.md)
}
```

### `Dynamic` — a runtime-erased wrapper for reflection

The reflection API needs a way to talk about values of unknown
static type. This RFC introduces a new predeclared type
`Dynamic` for that purpose. `Dynamic` is **not** a top type
(in the subtyping sense) — it is a wrapper. The mental model is
"a runtime box that has lost its static type"; static-typed
code stays static-typed, dynamic-typed values exist explicitly
inside `Dynamic`.

`Dynamic` is deliberately **separate** from the existing
predeclared `Any`. `Any` keeps its current spec wording and use
restriction (binding-boundary escape for variadic stdlib
bindings only; user-authored code never names it). The
existence of two distinct names is a cultural feature, not a
duplication: it lets us reject "let me put `Any` in my
function signature" without ambiguity, and it lets a `Dynamic`
spotted in user code be a deliberate, visible signal of "I am
talking to the reflection layer".

**Introduction.** A value of static type `T` becomes a
`Dynamic` value through **implicit widening, at exactly one
class of call site**: passing it as an argument to a function
in the `reflect` module whose corresponding parameter is
declared `Dynamic`. No other location widens. In particular:

- Returning a `T` from a function whose declared return type
  is `Dynamic` does **not** implicitly widen — declare the
  return as the concrete type or use `reflect.box` explicitly.
- Assigning a `T` to a `var x: Dynamic` does **not** widen
  unless the RHS is itself a `Dynamic`-typed expression.
- A `[]Dynamic` literal does **not** widen its elements — the
  user writes `[reflect.box(x), reflect.box(y)]`.

The single implicit-widening site keeps the ergonomics good
for the dominant use case (`reflect.typeOf(counter)`,
`reflect.fields(reflect.typeOf(counter))`) without leaking
the boxing into the general expression language.

**`reflect.box` for explicit boxing.** When the user wants a
`Dynamic` value outside a reflect call (e.g., to put into
`[]Dynamic`), they write `reflect.box(v)`. This is the same
operation as the implicit widening, just named.

**Elimination.** A value of type `Dynamic` cannot be used
where a concrete `T` is expected, with **no** implicit cast,
ever. The only operations on a `Dynamic` are functions in the
`reflect` module. To recover a concrete value from a
`Dynamic`, the user inspects through `reflect.kind` /
`reflect.typeOf` and uses `reflect.unbox<T>(d: Dynamic): Result<T>`
to attempt the unwrap (`Err` on type mismatch).

**Collections.** `[]Dynamic`, `Map<string, Dynamic>`, etc. are
allowed, but the spelling `Dynamic` appears in the type
annotation, so the user is visibly opting into runtime
erasure each time. There is no inference path that produces
a `Dynamic`-typed collection from concrete elements.

**Spellability.** `Dynamic` is spellable anywhere a type is
spellable — parameter types of user functions, return types,
field annotations, collection element types, type-parameter
arguments. Only the introduction and elimination rules above
are constrained. A user-authored `func cache(key: string,
default: Dynamic): Dynamic` is legal; the caller must produce
the `Dynamic` argument explicitly (`reflect.box(v)`), and the
callee must use `reflect.*` to act on it.

### Functions

```
// Total queries — always succeed.
func typeOf(v: Dynamic): Type
func typeName(t: Type): string         // e.g., "Counter", "int", "Option<int>"
func kind(t: Type): Kind

// Total queries — return [] for kinds where the question is
// vacuous (e.g., fields on a primitive). Use `kind(t)` to
// distinguish "no fields because it's not a record" from
// "record with zero fields".
func fields(t: Type): []FieldInfo
func methods(t: Type): []MethodInfo
func variants(t: Type): []VariantInfo
func typeArgs(t: Type): []Type

// Partial queries — return Result so the caller handles the
// failure explicitly. No panics. The Err carries a message
// shaped by the diagnostic codes in lang-spec/diagnostics.md.
func fieldValue(v: Dynamic, name: string): Result<Dynamic>   // Err: no such field, or v is not a class/record
func variantOf(v: Dynamic): Result<VariantInfo>              // Err: v is not a sum value
func elementType(t: Type): Result<Type>                      // Err: t has no single element type

// Boxing / unboxing
func box<T>(v: T): Dynamic              // explicit widen-to-Dynamic
func unbox<T>(d: Dynamic): Result<T>    // type-checked unwrap; Err on mismatch
```

**Failure discipline.** The reflection API is **panic-free** by
design. Every operation that can be wrong about its operand's
shape returns `Result`. This keeps the reflection contract
uniform with Tide's wider error-handling discipline (`Result`
on every fallible operation, no implicit exceptional control
flow) and avoids hidden non-local exits inside what reads as
ordinary getter calls. REPL surface effects benefit directly:
a bad `:inspect` expression cannot crash the session.

The API is deliberately minimal: enough to write a generic
pretty-printer, an interactive inspector, and the REPL's
`:type` / `:inspect`. Mutation through reflection
(`setField`, `callMethod`) is **not** in v1; it adds a much
larger semantics problem and unblocks no immediate user need.

### Use site sketch (informative)

```tide
import reflect

func dump(v: Dynamic): string {
  let t = reflect.typeOf(v)
  match reflect.kind(t) {
    Primitive => {
      // Recover a concrete value before reaching for stdlib
      // formatting. Each primitive arm narrows via
      // reflect.unbox<T> and uses its result as a real T.
      match reflect.typeName(t) {
        "int"    => match reflect.unbox<int>(v)    { Ok(n) => strconv.itoa(n), Err(_) => "<bad-int>" },
        "string" => match reflect.unbox<string>(v) { Ok(s) => "\"" + s + "\"", Err(_) => "<bad-string>" },
        "bool"   => match reflect.unbox<bool>(v)   { Ok(b) => if b { "true" } else { "false" }, Err(_) => "<bad-bool>" },
        // ... other primitives
        _        => "<primitive>",
      }
    },
    Class => {
      var out = reflect.typeName(t) + "{"
      for f in reflect.fields(t) {
        match reflect.fieldValue(v, f.name) {
          Ok(fv) => out = out + f.name + ": " + dump(fv) + ", ",
          Err(_) => out = out + f.name + ": <unreadable>, ",
        }
      }
      return out + "}"
    },
    // ...
  }
}
```

(The above is illustrative — not a normative example. `match`
on `Dynamic`-typed values requires the type rules in
`lang-spec/type-system.md` to admit it; the paired edit covers
this.)

### Performance and identity

This RFC makes no promises about `Type` value identity outside
equality. Whether `typeOf(a) == typeOf(b)` is `O(1)` is an
implementation concern. Two `Type` values *are* equal iff they
describe the same Tide type — this is the only externally
observable invariant.

## Prior art (informative)

Three neighbours were surveyed; this RFC's surface choices reflect
what was kept and what was rejected:

- **Deno REPL.** Source for `_` / `_error` last-value bindings.
  The "one prompt, no `:`-commands, multi-line via Ctrl-S"
  ergonomics were **rejected** as hostile to newcomers.
- **gore** (Go REPL). Source for the `:`-prefixed meta-command
  vocabulary (`:type`, `:show`, `:load`, `:reset`, plus
  `:write` as the "save session to file" exit-ramp), the
  `tide> ` / `... > ` continuation-prompt shape, and the
  brace-counting multi-line submission rule.
- **yaegi** (Go interpreter). Surveyed for completeness; its
  user-surface (no meta-commands, no history, plain-`fmt.Println`
  value printing) was **rejected** as too thin — the user
  experience reads as a debug shim, not a tool.

The competing Tide-adjacent language **Borgo** ships no REPL,
so there is no prior parity feature to chase. `tide repl` is a
differentiator.

## Alternatives considered

1. **Compile-time `typeof` operator instead of a runtime
   `reflect` module.** `typeof x` would yield a static type
   description, like TypeScript's `typeof`. *Rejected for v1.*
   Static `typeof` requires sema; sema is Phase 2. Static
   `typeof` also doesn't help the REPL pretty-printer, which
   has *values* in hand and wants to ask "what type is this *at
   runtime*?" — a different question.
2. **Reflection as a compiler magic-builtin set, not a
   user-importable module.** `__typename__(v)` style. *Rejected.*
   Tide is committed to the principle that things accessible to
   the language live in normal modules (D6). The binding
   surface already accommodates user-facing modules; `reflect`
   slots in naturally.
3. **Notebook-style cells instead of a line-based REPL.**
   *Rejected for v1.* Higher UI effort; does not change what
   the user can *do*. Can layer on top of the line-based REPL
   later.
4. **Auto-print without reflection — call `fmt.println(v)` for
   the value and accept Go-shaped output.** *Considered.* This
   is what a "REPL without reflection" looks like — and it's
   exactly the unsatisfying experience this RFC is meant to
   avoid. The cost of *not* having reflection is permanently
   visible in every interactive session.
5. **Reuse the existing `Any` for reflection parameters
   instead of introducing `Dynamic`.** *Rejected.* `Any` already
   exists as a binding-boundary escape type, restricted to
   variadic-stdlib bindings. Repurposing it as a user-facing
   dynamic-typing wrapper would (a) culturally signal "Tide is
   gradually-typed now" — exactly the wrong message — and (b)
   make it harder to refuse the next request ("if `Any` is
   user-facing in reflect, why can't I take it as a parameter
   in my own function?"). `Dynamic` as a separate token makes
   the cultural line speakable: `Any` is internal FFI, `Dynamic`
   is the runtime-erased wrapper used to talk to reflection.
6. **Explicit-only boxing — no implicit widening at any call
   site, including `reflect.*`.** *Considered.* Forces every
   reflection use to write `reflect.typeOf(reflect.box(x))`,
   which is honest but verbose. The single implicit-widening
   site — `reflect.*` parameter positions where the formal type
   is `Dynamic` — is the only place implicit widening fires,
   and `reflect.box` remains available for any other site
   (notably building a `[]Dynamic` literal). This is the
   minimum implicit-conversion budget that keeps the common
   case readable without leaking the boxing into the wider
   expression language.

## Paired edits

- `lang-spec/keywords.md` — add `Dynamic` to the predeclared-types
  list (alongside `bool`, `int`, ..., `Any`, `error`). The note
  on `Any` is **not** changed; `Dynamic` lives next to it as a
  separate, user-facing name.
- `lang-spec/grammar.ebnf` — add `Dynamic` to the predeclared
  identifier set used by `NamedType`. No new production needed
  (it lexes as an identifier).
- `lang-spec/type-system.md` — add the `Dynamic` type with:
  introduction rules (implicit widen at `reflect`-module call
  sites only; explicit via `reflect.box`); the no-narrowing
  rule (a `Dynamic` value is acted on only through `reflect`,
  with `reflect.unbox<T>` as the type-checked recovery path).
  Cross-reference `Any` to make the cultural-line non-confusion
  explicit: `Any` is internal FFI, `Dynamic` is the user-facing
  runtime-erased wrapper.
- `lang-spec/builtins.md` — describe the `reflect` module's
  surface: `Type`, `Kind`, `FieldInfo`, `MethodInfo`,
  `VariantInfo`, the type-introspection functions, and the
  boxing pair (`box<T>`, `unbox<T>`). Tag `reflect` as a
  runtime-supplied module (not a Go-stdlib binding); its
  metadata commitments are governed by the three D18
  invariants (private/public layer split, version locking,
  append-only ABI within a compatibility window).
- `lang-spec/diagnostics.md` — new diagnostic codes for the
  Err payloads returned by the partial reflection queries
  (`fieldValue` no-such-field / not-a-class, `variantOf`
  not-a-sum, `elementType` no-single-element, `unbox<T>`
  type-mismatch). No new panics are added — the reflection API
  is panic-free, so these are *return-shape* error codes, not
  raised diagnostics. New diagnostic codes for REPL-specific
  input rejections (top-level control-flow, user-typed `main`)
  remain raised diagnostics in the usual style.
- `README.md` — `tide repl` row in the entry-point table.

Note: the foundational decision "Tide has a runtime, and it is
part of the language contract" is **D18**, recorded in
`docs/design-decisions.md`. This RFC references D18 as the
already-established baseline; it does not re-establish it.

Runtime type-metadata layout is an implementation concern and
not normatively part of this RFC; the implementing PR will
document it in `docs/architecture.md`.

## Transition / compatibility

Purely additive on the language surface:

- `tide repl` is a new subcommand.
- `Dynamic` is a new predeclared type. Nothing in existing
  user code or spec mentions it.
- `reflect` is a new module.
- The existing `Any` predeclared identifier is **not** touched.
  Its current restriction (binding-boundary only,
  user-authored code never introduces it) stays as written in
  `lang-spec/type-system.md`. Keeping `Any` and `Dynamic`
  separate is a deliberate cultural-line measure (see the
  `Dynamic` section above).

No existing code breaks. The reflection API is unreachable from
code that does not `import reflect`, so programs that don't
opt into reflection observe Tide unchanged.

## Open questions

1. **Re-binding `let` / `var` at the prompt.** Typing
   `let x = 1` followed by `let x = "hi"` *shadows* the first
   (Rust-style: `x` is now a fresh binding of a different
   type). Decided: shadow, because shadowing is the natural
   REPL idiom and Tide's existing nested-scope semantics
   already permit `let` shadowing inside blocks. Closures
   captured by an earlier input still see the earlier `x`;
   new inputs see the new one. The implementing PR may
   surface edge cases (e.g. shadowing a `var` with a `let` of
   a different type while a prior closure mutates the
   original) — those land as point fixes, not contract
   changes.
2. **`:type <expr>` on a not-yet-bound name.** If the user
   types `:type x` and `x` is unknown, do we just say "no such
   binding" or try to parse the input as a fragment and report
   what *would* be its type? Bias: simple "no such binding".
   This rule also governs `:type _` and `:type _error` on a
   fresh session — neither has been bound yet, so both error
   with `repl: _ is unbound` / `repl: _error is unbound`.
3. **Reflection on closures / function values.** v1 says
   `kind(t) == Function` is possible but offers no further
   introspection (no `params`, no `return_type`). Closures
   make this messier still. Deferred until a user need lands.
4. **REPL behaviour for prior let-initialiser side effects.**
   If `let now = time.Now()` is typed once, subsequent inputs
   reading `now` should see the value from the original
   evaluation, not re-evaluate the initialiser. This is
   user-observable; the implementing PR must preserve this
   invariant regardless of internal strategy.
5. **Reflection mutation (`setField`, `callMethod`) deferred.**
   v1 reflection is read-only. Mutation by name introduces
   type-safety obligations the RFC is not ready to commit to
   (does the receiver narrow? can you set a `let` field?).
   Tracked here so a future RFC has the starting point.
